## What this is

SmartSlides: a native macOS SwiftUI slideshow app. Unlike a true-random slideshow, it generates one deterministic, seeded "timeline" of scenes up front and plays that same sequence start-to-finish (looping) until the user regenerates the hash. Personal/friends-only distribution — no App Sandbox, no App Store, ad-hoc signing.

## Build & run

The Xcode project is generated by [xcodegen](https://github.com/yonaskolb/XcodeGen) from `project.yml` — it is **not** hand-maintained. After adding, removing, or renaming any file under `SmartSlides/`, you must regenerate the project or Xcode won't see the change:

```bash
xcodegen generate
```

Build from the CLI:

```bash
xcodebuild -project SmartSlides.xcodeproj -scheme SmartSlides -configuration Debug build
```

Run the built app:

```bash
APP_PATH=$(find ~/Library/Developer/Xcode/DerivedData -name "SmartSlides.app" -path "*Debug*" | head -1)
open "$APP_PATH"
```

Or just `open SmartSlides.xcodeproj` and run from Xcode. There is no test target or lint config in this repo yet.

## Architecture

**Timeline-first, not live-random.** The core design constraint: `TimelineGenerator` builds the *entire* `[SlideScene]` sequence in one pass from a seed, and playback (`SlideshowPlayerViewModel`) just walks an index through that fixed array. Nothing during playback calls into the RNG. Same folder + same enabled layouts + same seed always reproduces the same timeline (see `SeededRandomNumberGenerator.swift`'s `SplitMix64`, since Swift's built-in `Random` is not stable across runs).

**Two windows, two view-models:**
- `SettingsViewModel` (`ViewModels/SettingsViewModel.swift`) owns the main settings window's state: folder selection, scanned `[ImageAsset]`, `AppSettings` (persisted), and timeline (re)generation. `SmartSlidesApp.swift` holds this as the single source of truth for the app's lifetime.
- `SlideshowPlayerViewModel` (`ViewModels/SlideshowPlayerViewModel.swift`) is created fresh each time `SlideshowWindowController` opens the fullscreen slideshow window, seeded from `SettingsViewModel.settings` (timeline, start index, durations). It owns the playback `Timer`, pause/resume, and next/prev. Its `onIndexChange` callback writes the current index straight back into `SettingsViewModel` so pause/resume/relaunch always land on the same slide.

**Timeline generation (`Services/TimelineGenerator.swift`):** shuffles portrait and landscape asset pools independently (seeded), then walks each pool deciding *per scene* — via a coin-flip on the same seeded RNG — whether to emit a paired side-by-side scene or a single-image scene, when both the pair and single layout for that orientation are enabled. This is deliberate: it used to greedily pair-until-a-leftover, which meant single-image layouts almost never appeared even when enabled. Any layout, folder, or subfolder-toggle change generates a **new** seed before regenerating (`SettingsViewModel.regenerateTimelineWithNewSeed`), since the old seed's decisions don't carry meaning against a changed image pool/layout set. Odd leftovers fall back to the single-image layout if enabled, otherwise get skipped and counted in `skippedCount`.

**Two-image layout sizing (`Views/SlideshowView.swift`'s `PairedImageView`):** side-by-side scenes are *not* two independently-aspect-fit images in fixed half-width columns (that produces mismatched heights and a black gap between them). Instead it computes one shared height from both images' real aspect ratios — `combinedHeight = min(containerHeight, containerWidth / (aspect1 + aspect2))` — then sizes each image to exactly that height with no extra letterboxing, so the pair sits flush together and landscape pairs share the same height. Same logic conceptually mirrored (at thumbnail scale) in `Views/TimelineStripView.swift`'s `SceneThumbnailCell`.

**Fullscreen playback window (`Controllers/SlideshowWindowController.swift`):** uses a custom `KeyableBorderlessWindow` subclass overriding `canBecomeKey`/`canBecomeMain`. This matters — a stock borderless `NSWindow` silently cannot become key, so it never receives *any* keyboard events (this was a real bug: Escape/arrows/space did nothing until this override was added). Keyboard handling itself lives in `Views/SlideshowView.swift`'s `KeyEventCatcher`, an `NSViewRepresentable` that becomes first responder and maps key codes 49/124/123/53 (space/right/left/esc).

**Thumbnails (`Services/ThumbnailStore.swift`):** `@MainActor` `ObservableObject` that generates small downsampled thumbnails via `CGImageSourceCreateThumbnailAtIndex` (not full image decodes) concurrently via `withTaskGroup`, keyed by URL and cached in memory for the app's lifetime. Regenerating the timeline calls `SettingsViewModel.refreshThumbnails()`, which diffs against the cache and only generates for new URLs; in-flight generation is cancelled if the timeline changes again before it finishes. `TimelineStripView` shows a progress state while `isGenerating` is true.

**Persistence (`Services/AppSettingsStore.swift`):** `AppSettings` (including the generated `timeline` and `currentTimelineIndex`) is written as JSON to `~/Library/Application Support/SmartSlides/settings.json` on essentially every state change. Folder access survives relaunch via a security-scoped bookmark (`NSOpenPanel` → `bookmarkData(options: .withSecurityScope)` → `resolveBookmark` on launch calls `startAccessingSecurityScopedResource()`). If the bookmark fails to resolve, `SettingsViewModel.needsFolderReselect` flips true and the UI should prompt for reselection rather than crash.

## Conventions specific to this repo

- No sandbox entitlements, ad-hoc code signing (`CODE_SIGN_IDENTITY: "-"` in `project.yml`) — intentional, this ships informally to friends, not the App Store.
- Layout logic assumes exactly 4 fixed layouts (`SlideLayout`: `onePortrait`, `oneLandscape`, `twoPortraitsSideBySide`, `twoLandscapesSideBySide`). No mixed portrait+landscape pairing, no Ken Burns/pan/zoom, no 4-image layouts — these are explicitly out of scope for this MVP.
- Orientation classification is aspect-ratio-only: `height > width` → portrait, else landscape (square counts as landscape) — see `ImageAsset.swift`.
